# WhatsApp Operator Flow

## Purpose

The WhatsApp operator is a lightweight operational console for hiring interviews. It should help an operator answer three questions without opening HiringOS:

1. Which interviews are scheduled?
2. Which interviews have been completed?
3. Which candidates should be prioritized and followed up?
4. What happened in a completed interview?

The operator should not jump directly from a greeting to an interview analysis. It should first show lists, then let the user choose an interview, then show the report and suggested follow-up questions.

## Operating Principle

Determinism owns facts and navigation. The LLM sits at the edge.

- Database queries decide whether scheduled or completed interviews exist.
- List responses are deterministic and generated by code.
- Empty states are deterministic and generated by code.
- Menu/navigation fallbacks are deterministic and generated by code.
- The LLM is used only after a specific interview has been selected, or when answering a question grounded in that interview's transcript and structured intelligence.
- The system must never infer that the operator wants the latest interview report merely because they tapped `Completed`, `Shortlist`, `Follow-ups`, `Ask`, or `View result`.
- Product-facing copy is English-only. Internal collaboration can happen in Spanish, but WhatsApp responses, buttons, prompts, metadata, and landing copy must remain English.

## Conversational UX

Messages should feel like a lightweight operating console, not raw database output.

- Use WhatsApp-friendly bold with `*...*` for section labels and high-signal values.
- Use short bullet points for scanability.
- Use only a small emoji set, sparingly:
  - `📅` schedule/date
  - `🕒` time
  - `✅` completed/positive signal
  - `⚠️` risk/red flag
  - `🎙️` voice note support
- Include one short prompt section in useful places, such as `Try asking: ...`.
- Mention voice notes in the opening experience: the operator may type or send a voice note.
- Avoid decorative or excessive emoji use. The tone is operational and polished.

## Architecture Contract

The WhatsApp provider is transport only. Infobip-specific payload parsing and outbound message delivery live in `whatsapp-operator`.

Conversation state, business decisions, Supabase queries, ranking, follow-up grouping, report generation, and available actions live in the Hiring Operator Orchestrator:

- Shared module: `supabase/functions/_shared/hiring-operator-orchestrator.ts`
- Direct test endpoint: `supabase/functions/hiring-operator-orchestrator`

The adapter sends a normalized event to the orchestrator:

```json
{
  "channel": "whatsapp",
  "user": { "phone": "573001112233" },
  "message": {
    "type": "text",
    "text": "shortlist",
    "action": null
  },
  "context": {
    "tenant": "retail-ai",
    "locale": "en-US"
  }
}
```

The orchestrator returns provider-neutral text and actions:

```json
{
  "state": "shortlist",
  "text": "Hiring shortlist: ...",
  "actions": [
    { "id": "best_pick", "label": "Best Pick" },
    { "id": "compare", "label": "Compare" },
    { "id": "followups", "label": "Follow-ups" }
  ]
}
```

Infobip buttons are only a rendering of `actions`. The flow must continue to work when the operator types `completed`, `shortlist`, `followups`, or `menu` manually.

## Scheduled Interview Dispatch

Scheduled calls are persisted in `interview_session` with `status = draft`, `intent = schedule_call`, `intake_source = landing`, and `scheduled_at`.

The Edge Function `dispatch-scheduled-interviews` is the automatic scheduler target. It:

- Finds due scheduled interviews where `scheduled_at <= now()`.
- Calls `start-interview-call` for each due interview.
- Leaves future interviews untouched.
- Returns counts for due, started, and failed interviews.

The database cron job is defined in:

`supabase/migrations/20260522222000_schedule_dispatch_scheduled_interviews.sql`

It should run every minute through `pg_cron + pg_net`.

## Core States

## Button Contract

Buttons are navigation only. They must never require hidden state.

- Main Menu: `Scheduled`, `Completed`, `Shortlist`
- Scheduled List: `Completed`, `Shortlist`, `Menu`
- Completed List: `Shortlist`, `Best Pick`, `Compare`
- Previous List: `Completed`, `Shortlist`, `Menu`
- Shortlist: `Best Pick`, `Compare`, `Follow-ups`
- Best Pick: `Summary`, `Risks`, `Phone`
- Compare: `Best Pick`, `Shortlist`, `Menu`
- Follow-ups: `Best Pick`, `Shortlist`, `Menu`
- Selection Required: `Completed`, `Shortlist`, `Menu`
- Interview Report: `Summary`, `Risks`, `Phone`
- Candidate Q&A: `Summary`, `Risks`, `Phone`

The operator selects an interview by replying with the item number, candidate name, or phone last four digits. There is no global `Ask` button and no global `View result` button in the MVP because both require a selected interview. `Summary`, `Risks`, and `Phone` are valid only after a deterministic candidate selection, such as tapping `Best Pick`, selecting a numbered candidate, or naming a candidate in the question. Free-form candidate questions are supported only when the question identifies a candidate or follows a deterministic selection.

### 1. Home / Main Menu

Triggered by:
- `hello`
- `hi`
- `start`
- `menu`
- `back`

Response:
- Greet the operator.
- Briefly explain that Hiring OS can show scheduled interviews, completed interviews, and interview reports.
- Include live counts from real scheduled interviews and real completed interviews.
- Show only the primary navigation options.
- Offer actions:
  - View scheduled interviews
  - View completed interviews
  - View hiring shortlist

### 2. Scheduled Interviews List

Triggered by:
- `scheduled`
- `schedule`
- `today`
- `refresh`
- `scheduled_today`

Response:
- Show scheduled interviews as a list.
- Source only real `interview_session` rows with `status = draft`, `intent = schedule_call`, and a non-null future `scheduled_at`.
- Each item should include candidate name, scheduled date/time, phone, and status.
- If none are scheduled, say no interviews are scheduled and invite the operator to check back.
- Offer actions:
  - View completed interviews
  - View hiring shortlist
  - Return to menu

### 3. Completed Interviews List

Triggered by:
- `completed`
- `done`
- `finished`
- `processed`
- `completed_today`

Response:
- Show available completed interviews as a list, not a full analysis.
- Each item should include candidate name, phone, completion date/time, and result when available.
- Include the primary next step when available.
- If no completed interviews exist, say so and offer menu or scheduled interviews.
- Offer actions:
  - Select/view a listed interview
  - View hiring shortlist
  - View follow-up queue
  - Return to menu

### 4. Previous Interviews List

Triggered by:
- `previous`
- `past`
- `recent`
- `history`
- `previous_interviews`

Response:
- Show completed interviews from before today.
- Each item should include candidate name, phone, completion date/time, and result when available.
- Offer actions:
  - Select/view a listed interview
  - Return to completed interviews
  - View hiring shortlist
  - Return to menu

### 5. Hiring Shortlist

Triggered by:
- `shortlist`
- `candidate insights`
- `top candidates`
- `best candidates`
- `strongest candidates`
- `who should we hire`

Response:
- Rank completed candidates from real interview intelligence.
- Use `signal_indicators.overall_result` when available.
- Fall back to `operational_interest.fit_quality` and `immediate_hire_potential` for older records.
- Include candidate name, visible result, primary positive signal, and next step.
- Do not invent candidates or results.
- Offer actions:
  - Choose the best pick
  - Compare candidates
  - View follow-up queue

### 6. Best Pick

Triggered by:
- `best pick`
- `best`
- `top pick`
- `who would you pick`
- `who is the strongest candidate`
- `best_pick`

Response:
- Select the highest-ranked real candidate from the shortlist.
- Persist that candidate as the current interview context.
- Explain why this candidate is first using only real interview intelligence.
- Include the primary next step.
- Offer actions:
  - Show summary
  - Show risks
  - Show phone

### 7. Candidate Comparison

Triggered by:
- `compare candidate 1 and candidate 2`
- `compare [candidate name] and [candidate name]`
- `candidate 1 vs candidate 2`
- `compare`

Response:
- If two candidates are identified, compare only those two candidates using structured interview intelligence and transcript evidence.
- If the request says only `compare`, ask the operator to name two candidates or use list numbers from the last shortlist/completed list.
- Offer actions:
  - Choose the best pick
  - Return to shortlist
  - Return to menu

### 8. Follow-Up Queue

Triggered by:
- `followups`
- `follow-ups`
- `next steps`
- `to do`
- `verify queue`

Response:
- Group real next steps from `operational_interest.next_steps_required` and `operational_interest.next_steps`.
- Show each action with the candidates that need it.
- Fall back to red-flag review only when no next steps exist.
- Offer actions:
  - Choose the best pick
  - View shortlist
  - Return to menu

### 9. Interview Report

Triggered by:
- Selecting an interview from a list
- Asking for the result of a specific candidate
- `summary`
- `summarize`
- `report`

Response:
- Show the selected interview report.
- Include:
  - Candidate name
  - Phone
  - Interview date/time
  - Overall result
  - Summary notes
  - Positive signals
  - Risks / red flags
  - Suggested next step
- End with suggested questions:
  - What were this candidate's strengths?
  - Any red flags?
  - What should we verify next?
- Offer actions:
  - Show summary
  - Show risks
  - Show phone

### 10. Candidate Phone

Triggered by:
- `phone`
- `phone number`
- `contact`
- `what is [candidate name]'s phone number`

Response:
- Return only the real phone number stored with the selected or named candidate.
- If no candidate is selected or named, ask the operator to select a candidate first.
- Offer actions:
  - Show summary
  - Show risks
  - Show phone

### 11. Candidate Risks

Triggered by:
- `risks`
- `red flags`
- `concerns`
- `what should we verify before hiring [candidate name]`

Response:
- Show only risks, red flags, uncertainty, and verification needs from structured interview intelligence and transcript evidence.
- If no candidate is selected or named, ask the operator to select a candidate first.
- Offer actions:
  - Show summary
  - Show risks
  - Show phone

### 12. Candidate Q&A

Triggered by:
- Free-form questions after a deterministic candidate selection
- Questions mentioning a candidate name or phone

Response:
- Answer only from transcript and structured interview intelligence.
- If evidence is missing, say so.
- Keep answers brief and operational.

## Selection Rules

For MVP validation, list items can be referenced by:
- Candidate first name or full name
- Last four digits of phone
- Interview ID prefix
- A simple numbered list item, if present in the previous message

When the user asks for a result without specifying a candidate, show the completed-interviews selection list instead of choosing an interview implicitly.

## Data Sources

Scheduled interviews:
- MVP demo uses a deterministic mock schedule in `whatsapp-operator`.
- The mock schedule includes two interviews for May 22, 2026 after 3 PM and three interviews for May 23, 2026.
- Future production source should be the scheduling system or `interview_session.scheduled_at`.

Completed interviews:
- `interview_session`
- Has `transcript` or `interview_intelligence_b1`
- Status may be `awaiting_evaluation` or `evaluated`

Interview report:
- `interview_session.interview_intelligence_b1`
- `interview_session.transcript`
- Candidate identity fields from `interview_session`

## Current MVP Constraints

- The WhatsApp flow is English-only.
- Infobip sends inbound events to one Supabase Edge Function.
- Retell completion data may arrive by webhook or be synced by `sync-retell-call`.
- The flow should prioritize clarity over exhaustive reporting.
